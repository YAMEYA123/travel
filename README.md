# 2026 欧洲旅行网页应用

同时服务于**行前规划**和**旅途中执行**的 React 旅行应用，覆盖阿姆斯特丹、科隆、慕尼黑、萨尔茨堡和维也纳。行前用于连续检查11天行程、转场强度、住宿票务与准备进度；旅行中用于查看当天时间轴、下一段交通、今晚住宿、费用分账和应急资料。

## 信息架构

- **总览**：可切换“规划全程 / 查看今天”，集中展示路线、准备进度、下一段交通和紧急事项
- **行程**：可切换“全程规划 / 单日执行”，兼顾行前比较和现场时间轴
- **预订**：按待处理、全部、已确认筛选交通、住宿和景点记录；待办使用空复选框，手动完成显示绿色勾，已确认订单固定显示蓝色票券勾
- **工具**：费用预算、多人分账、天气应急、行李、App、语言、购物退税、签证和风险

## 地图与紧凑交互

- 每日地图是核心功能：D0–D10的活动点、酒店点、顺序编号和交通线路必须与时间轴同步
- 地图使用仓库内置Leaflet；默认CartoDB Voyager底图，可手动切换高德备用底图
- 点击时间轴定位地图点，点击地图点高亮对应活动；地图不可用时仍保留地点与外部导航
- 活动类型使用固定颜色和图标：火车蓝、地铁紫、公交橙、步行绿、景点红、餐饮琥珀、住宿紫
- 手机端采用固定应用壳、地图工作区和紧凑列表；详细说明按需打开，不在主页面连续铺开
- 预订和工具使用列表—详情模式，避免一个页面堆叠全部内容
- 预订列表必须明确区分“待办勾选”“已确认订单”和“打开详情”三种语义：已确认订单不可继续显示为待勾选状态

移动端底部只保留四个一级入口；工具内容使用可展开分组，不使用多层横向页签。旧的 `#/days`、`#/route` 和 `#/budget` 地址会自动映射到新的页面。

## 本地运行

需要 Node.js 22 或更高版本：

```bash
npm install
npm run dev -- --port 4173
```

生产构建：

```bash
npm run typecheck
npm run build
npm run preview -- --port 4174
```

请勿使用 `8080` 端口；该端口属于 Loomi 生产后端。

## 项目结构

```text
src/
├── App.tsx       # 四个一级入口、双状态首页与行程双视图
├── data.ts       # 行程、预订和路线的唯一数据源
├── mapData.ts    # D0–D10地图中心、酒店与活动坐标
├── TripMap.tsx   # Leaflet地图、线路与列表双向联动
├── ExpenseLedger.tsx # 本地多人分账、结算与导出
├── Toolbox.tsx   # 按场景展开的实用信息、清单、App、购物、语言、签证与风险
├── toolboxData.ts # 工具箱结构化数据
├── styles.css    # 铁路控制台视觉系统与响应式布局
├── toolbox.css   # 工具目录、清单与移动端分账样式
├── types.ts      # TypeScript 数据类型
└── main.tsx      # 应用入口
```

旧版单文件保留在 `legacy/europe-travel-2026.html`，便于对照和回滚。根目录的 `europe-travel-2026.html` 是已构建的新版应用入口，因此原网址保持不变。

## 数据与隐私

- 活动、预约、行李和签证勾选状态仅保存在浏览器 `localStorage`
- 多人分账成员和消费记录保存在本机，并可导出JSON备份
- 生产版注册Service Worker，首次访问后可缓存核心页面和已访问资源
- 应用不保存姓名、护照号码和订单确认号
- 修改交通或酒店时优先编辑 `src/data.ts`，并同步正式文字版攻略

## 响应式基线

- 移动端基准宽度：360、390、430px
- 平板基准：768px
- 桌面基准：1280、1440px
- 最小触控区域：44×44px
- 底部导航适配 `safe-area-inset-bottom`
- 请不要在移动端增加超过四个一级导航项，也不要重新引入嵌套横向页签

## 部署

仓库沿用 GitHub Pages 的 `main` 分支静态发布方式。发布前运行 `npm run build`，再将 `dist/index.html` 复制为根目录的 `europe-travel-2026.html`，并同步 `dist/assets/`。Vite 的 `base` 已配置为 `/travel/`。
