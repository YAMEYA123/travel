# 荷德奥旅游手账

同时服务于**行前规划**和**旅途中执行**的 React 旅行应用，覆盖阿姆斯特丹、科隆、慕尼黑、萨尔茨堡、奥地利湖区和维也纳。行前用于连续检查11天行程、转场强度、住宿票务与准备进度；旅行中用于查看当天时间轴、下一段交通、今晚住宿、费用分账和应急资料。

## 视觉方向

应用采用“**清爽旅行手账**”视觉：以湖水青、珊瑚橙、森林绿和纸张白建立轻快的欧洲旅行氛围，同时保留地图探险类应用的清晰结构与紧凑信息密度。

- 地图、时间轴和底部导航保持工具属性，不使用纯装饰性的海报布局
- 邮戳、虚线路线、票根缺口和纸张纹理只用于标题、住宿、预订状态等少量关键位置
- 活动卡按交通、景点、餐饮、住宿和提醒使用独立颜色，不能退化为连续白色表格行
- 工具子页必须保持同一套旅行手账语言：使用场景页头、分类图标、进度与色彩编码，不能只呈现无差别文字列表
- 国家主题色继续承担导航意义：荷兰橙、德国蓝、奥地利绿
- 移动端一屏优先展示地图、当天主要节点与今晚住宿，避免为了装饰制造额外滚动
- 保持正文高对比度、44px触控基线和减少动态效果支持

## 信息架构

- **总览**：可切换“规划全程 / 查看今天”，集中展示路线、准备进度、下一段交通和紧急事项
- 首页以地图作为视觉主体，充分使用中下部空间；移动端同步提高地图可视高度，减少额外模块对行程信息的挤压
- 移动端首页信息区采用固定统计行与自适应内容行，避免统计卡和交通、待办卡片之间出现突兀空隙
- 添加到手机主屏幕后使用专属旅行手账图标，并同时提供 Apple Touch Icon 与 PWA 192/512 像素图标
- **行程**：可切换“全程规划 / 单日执行”；全程规划支持逐日或全部展开，并用独立按钮进入对应单日地图
- **预订**：状态按待办、全部、已确认筛选，内容再按全部、住宿、景点、交通拆分；租车、驾照材料和车票统一归入交通。待办使用空复选框，手动完成显示绿色勾，已确认订单固定显示蓝色票券勾
- **工具**：按“出发前准备 / 旅途中使用”组织费用、行李、签证、App、应急、语言、购物退税和风险入口

工具子页采用渐进展开而非连续平铺：行李与签证默认打开首个未完成分组，应急与购物按主题折叠，地名语言默认展示常用项并支持国家筛选。行程风险使用高／中／低等级筛选，当前等级内直接展示风险标题与应对方案，不再为每条风险增加下拉层级。移动端折叠标题保持横排，并与清单项复选框样式隔离，避免窄屏逐字换行。

常用 App 使用随站点发布的本地品牌图标，避免境外图标服务在旅途中加载缓慢；图标异常时自动回退到名称缩写。页面按用途分组为轻量图标启动器，点击整张图标卡即优先启动已安装 App 或官方通用链接，不再叠加“打开 App”按钮；安装/更新只保留为图标下方的弱化商店入口。Android 使用应用包名直接启动并在未安装时回退 Google Play，iOS 仅对有公开可靠 URL Scheme 的应用使用深链。

地名语言覆盖行程、地图、住宿和预订中涉及的主要地点；荷兰段提供中文、荷兰语与英文，德国和奥地利段提供中文、德语与英文，并支持跨语言搜索和复制当地名称。

荷兰公共交通详情标注 OVpay 支付规则：支持非接触式 Visa、Mastercard 及手机钱包，每位乘客使用自己的卡，并用同一张实体卡或同一台设备完成刷入、刷出；不同运营商之间换乘时按读卡器提示分别出站、进站。已购国际列车票等不适用 OVpay 的路段会单独注明。

## 地图与紧凑交互

- 每日地图是核心功能：D0–D10的活动点、酒店点、顺序编号和交通线路必须与时间轴同步
- 地图使用仓库内置Leaflet；默认CartoDB Voyager底图，可手动切换高德备用底图
- 点击时间轴定位地图点，点击地图点高亮对应活动；外部地图入口统一放在活动详情底部，避免地图上出现重复悬浮导航
- 详情弹窗与遮罩使用独立模态层级，必须覆盖地图图例、导航按钮、Leaflet控件及地图全屏层
- 点击每日活动会打开完整详情，包含地点简介、推荐看点、现场提示、时间费用和当天风险提醒
- 活动类型使用固定颜色和图标：火车蓝、自驾深青、地铁紫、公交橙、步行绿、景点红、餐饮琥珀、住宿紫
- 手机端采用固定应用壳、地图工作区和紧凑列表；详细说明按需打开，不在主页面连续铺开
- 预订和工具使用列表—详情模式，避免一个页面堆叠全部内容
- 预订列表必须明确区分“待办勾选”“已确认订单”和“打开详情”三种语义：已确认订单不可继续显示为待勾选状态

移动端隐藏桌面侧栏及顶部品牌栏，仅保留底部四个一级入口，以释放完整视口高度；工具内容使用可展开分组，不使用多层横向页签。旧的 `#/days`、`#/route` 和 `#/budget` 地址会自动映射到新的页面。

工具详情页的内容区独立承担纵向滚动；移动端工作区必须继承应用壳扣除顶部、底部导航后的可用高度，避免 `100dvh` 嵌套导致列表末端无法触达。

## 本地运行

需要 Node.js 22 或更高版本：

```bash
npm install
npm run dev -- --port 4173
```

生产构建：

```bash
npm run typecheck
npm run build
npm run preview -- --port 4174
```

请勿使用 `8080` 端口；该端口属于 Loomi 生产后端。

## 项目结构

```text
src/
├── App.tsx       # 四个一级入口、双状态首页与行程双视图
├── data.ts       # 行程、预订和路线的唯一数据源
├── activityGuides.ts # 每日目标地简介、看点与现场提示
├── mapData.ts    # D0–D10地图中心、酒店、停车点与活动坐标
├── TripMap.tsx   # Leaflet地图、线路与列表双向联动
├── ExpenseLedger.tsx # 本地多人分账、结算与导出
├── Toolbox.tsx   # 按场景展开的实用信息、清单、App、购物、语言、签证与风险
├── toolboxData.ts # 工具箱结构化数据
├── styles.css    # 清爽旅行手账视觉系统与响应式布局
├── toolbox.css   # 工具目录、清单与移动端分账样式
├── types.ts      # TypeScript 数据类型
└── main.tsx      # 应用入口
```

旧版单文件保留在 `legacy/europe-travel-2026.html`，便于对照和回滚。根目录的 `europe-travel-2026.html` 是已构建的新版应用入口，因此原网址保持不变。

## 数据与隐私

- 活动、预约、行李和签证勾选状态仅保存在浏览器 `localStorage`
- 多人分账成员和消费记录保存在本机，并可导出JSON备份
- 每个预订详情可直接创建关联费用记录，自动带入预订名称与费用类别；分账记录支持反向打开对应预订
- 分账支持欧元、人民币、美元以及 IHG、万豪、希尔顿积分；酒店积分单独累计，不混入现金结算
- 付款人可选择“各自支付（总额均分）”；录入两人总额后，个人现金支出自动按总额的 1/2 计入
- 生产版注册Service Worker，预缓存入口及当前构建的JS/CSS；离线时只为页面导航回退HTML，避免把HTML错误返回给脚本或样式请求
- 应用不保存姓名、护照号码和订单确认号
- 修改交通或酒店时优先编辑 `src/data.ts`，并同步正式文字版攻略

## 响应式基线

- 移动端基准宽度：360、390、430px
- 平板基准：768px
- 桌面基准：1280、1440px
- 移动端顶部页签导航统一为168px总宽，组内按钮按页签数量等分
- 行程页头使用“可收缩日期导航 + 固定页签”网格，390px及更窄视口不得裁切“单日 / 全程”
- 行程、预订和工具页的移动端页眉统一为53px高度与10px水平内边距
- 所有一级页面页眉禁止在纵向Flex布局中收缩：桌面固定64px、移动端固定53px，长列表不得挤压页眉
- 全程行程的 D0–D10 圆形序号必须在桌面端和移动端保持水平、垂直居中
- 最小触控区域：44×44px
- 底部导航适配 `safe-area-inset-bottom`
- 请不要在移动端增加超过四个一级导航项，也不要重新引入嵌套横向页签

## 部署

仓库沿用 GitHub Pages 的 `main` 分支静态发布方式。发布前运行 `npm run build`，再将 `dist/index.html` 复制为根目录的 `europe-travel-2026.html`，并同步 `dist/assets/`。Vite 的 `base` 已配置为 `/travel/`。
